PART 0: CORE RULES (위반 불가)
==================================================

Rule 1: Bot은 판단하지 않는다. 계산하고 실행한다.
Rule 2: AI는 실행하지 않는다. 분석하고 추천한다.
Rule 3: AI 추천의 자동 반영은 금지한다. 반드시 22B 승인.
Rule 4: Safety > Risk > Strategy > Execution > Opportunity.
Rule 5: 리스크 정책은 어떤 시그널/추천보다 상위이다.
Rule 6: 모든 주문은 결정 로그를 남긴다 (Audit Trail).
Rule 7: 거래소 실포지션과 로컬 DB는 주기적으로 대조한다.

PART 2: REGIME DETECTION — 정량 판정표
==================================================

레짐 "판정"은 Bot이 rule-based로 수행.
AI는 판정 결과 위에 "해석/추천"만 수행.
같은 입력 → 같은 레짐 출력을 보장.

--------------------------------------------------
2.1 레짐 판정 규칙
--------------------------------------------------

BTC_BULLISH:
  조건 (모두 충족):
    - BTC 4H EMA50 상방
    - BTC 24H 수익률 > 0%
    - 펀딩레이트 < 0.05% (과열 아님)
  무효 조건:
    - 24H ATR/price > 5% (고변동성으로 전환)

BTC_BEARISH:
  조건 (모두 충족):
    - BTC 4H EMA50 하방
    - BTC 24H 수익률 < -1%
  무효 조건:
    - 급반등 (1H 수익률 > 3%)

BTC_SIDEWAYS:
  조건:
    - BTC 24H 수익률 -1% ~ +1%
    - 24H ATR/price < 3%
    - BB bandwidth < 평균의 70%

ALT_ROTATION:
  조건:
    - BTC 레짐 = SIDEWAYS 또는 BULLISH
    - BTC.D 24H 변화 < -0.3%
    - 상위 10 알트 평균 거래량 > 전주 평균의 1.5x

HIGH_VOLATILITY:
  조건:
    - 24H ATR/price > 5%
    - 또는 1H 가격변동 > 3%

LOW_VOLATILITY:
  조건:
    - 24H ATR/price < 2%
    - BB bandwidth < 평균의 50%

EVENT_RISK:
  조건:
    - FOMC/CPI 등 매크로 이벤트 24시간 이내
    - 또는 거래소 점검/장애 감지
  참고: 이벤트 캘린더는 수동 입력 또는 API

복합 레짐:
  - 두 레짐이 동시 충족 시 상위 우선:
    EVENT_RISK > HIGH_VOLATILITY > 나머지
  - UNKNOWN: 어떤 레짐도 충족하지 않으면
    → 전략 confidence 30% 감산, 신규 LIVE 진입 차단

--------------------------------------------------
2.2 레짐별 전략 허용표
--------------------------------------------------

  레짐             허용 전략 카테고리          차단
  ──────────       ─────────────────           ──────
  BTC_BULLISH      trend, breakout, multi      -
  BTC_BEARISH      mean_reversion, hedge       trend long 차단
  BTC_SIDEWAYS     mean_reversion, range       -
  ALT_ROTATION     regime, trend (알트만)      -
  HIGH_VOL         volatility, micro           신규 진입 축소 50%
  LOW_VOL          breakout (squeeze)          momentum 차단
  EVENT_RISK       -                           전체 신규 진입 차단
  UNKNOWN          -                           LIVE 진입 차단

--------------------------------------------------
2.3 AI의 역할 (레짐 관련)
--------------------------------------------------

  AI는 Bot의 레짐 판정 결과를 읽고:
  - "왜 이 레짐인지" 설명 (근거 지표 3개)
  - "다음 레짐 전환 가능성" 해석
  - "이 레짐에서 추천하는 전략 변경" 제안
  - Dashboard 패널 6에 출력

  AI가 하지 않는 것:
  × 레짐 판정 자체를 변경
  × 레짐 판정을 우회하여 전략 활성화

PART 3: ORDER & POSITION STATE MACHINE
==================================================

--------------------------------------------------
3.1 시그널 → 주문 상태 흐름
--------------------------------------------------

  SIGNAL_CREATED
    → RISK_CHECKED (pass / fail)
    → ORDER_SUBMITTED
    → PARTIALLY_FILLED / FILLED / REJECTED
    → SL_ATTACHED
    → TP_ATTACHED
    → MONITORING
    → TP_HIT / SL_HIT / TRAILING_HIT / MANUAL_CLOSE / TIME_EXIT
    → CLOSED
    → RECONCILED (거래소 대조 완료)

  각 전환마다 타임스탬프 + 사유 기록.

--------------------------------------------------
3.2 Order Idempotency (중복 주문 방지)
--------------------------------------------------

  규칙:
  - 같은 signal_id로 주문은 1회만 허용
  - 주문 전 DB에서 signal_id 존재 여부 확인
  - 주문 성공 후 즉시 DB 기록
  - DB 기록 실패 시 → 주문 상태를 ORPHANED로 마킹
    → 다음 reconciliation에서 처리

--------------------------------------------------
3.3 State Reconciliation (상태 대조)
--------------------------------------------------

  주기: 5분마다

  프로세스:
  1. Binance API로 실제 오픈 포지션 조회
  2. 로컬 DB의 오픈 포지션과 비교
  3. 불일치 발견 시:
     - DB에 있는데 거래소에 없음 → CLOSED로 마킹 + 경고
     - 거래소에 있는데 DB에 없음 → ORPHANED 포지션 생성 + 긴급 알림
     - 수량 불일치 → 로그 + 경고
  4. 결과를 Dashboard에 표시

--------------------------------------------------
3.4 Restart Recovery (재시작 복구)
--------------------------------------------------

  미니PC 재부팅 또는 프로세스 재시작 시:

  1. DB에서 마지막 상태 로드
  2. 거래소 API로 현재 포지션 조회
  3. Reconciliation 즉시 실행
  4. 불일치 있으면 → OBSERVE 모드로 시작 + 알림
  5. 불일치 없으면 → 이전 모드로 복귀
  6. 미체결 주문 확인 → 유효하면 유지, 만료면 취소

--------------------------------------------------
3.5 Kill Switch
--------------------------------------------------

  트리거 (자동):
  - 일일 손실 한도 초과
  - 거래소 API 3회 연속 실패
  - Reconciliation 불일치 감지
  - 예상 밖 포지션 발견

  트리거 (수동):
  - Dashboard KILL SWITCH 버튼
  - Telegram /kill 명령

  동작:
  - 즉시 신규 진입 차단
  - 미체결 주문 전부 취소
  - 기존 포지션은 유지 (SL이 보호)
  - 시스템 모드 → BLOCKED
  - 22B에게 긴급 알림

--------------------------------------------------
3.6 Audit Trail (결정 로그)
--------------------------------------------------

  모든 주문에 다음 정보 기록:

  signal_id:        시그널 고유 ID
  strategy_id:      전략 이름
  regime_snapshot:   주문 시점 레짐 + 근거 지표값
  risk_check_result: 통과한 리스크 룰 목록
  order_params:     주문 타입, 수량, 가격
  execution_result:  체결 가격, 수량, 수수료
  decision_reason:   "EMA cross + RSI 52 + Vol 1.8x"

  이 로그는 불변. 수정/삭제 불가.

PART 4: STRATEGY VALIDATION — 3단계
==================================================

기존 v1.1의 2단계를 3단계로 강화.

--------------------------------------------------
4.1 Stage 1: Stat Validated
--------------------------------------------------

  최소 표본:
  - 고빈도 전략 (일 1회+ 시그널): 50건+
  - 저빈도 전략 (주 2~3회 시그널): 30건+

  필수 지표:
  - Profit Factor >= 1.3
  - Win Rate >= 40%
  - MDD < 12% (15%에서 하향)
  - Expectancy > 0
  - Avg Win / Avg Loss >= 1.2

  통과 시 → Stage 2로 진행

--------------------------------------------------
4.2 Stage 2: Shadow Live
--------------------------------------------------

  실시간 시장 데이터로 주문 직전까지 계산.
  주문만 실행하지 않음.

  기간: 최소 1주

  추가 검증:
  - 수수료 반영 후 성과 유지 여부
  - 예상 체결가 vs 실제 시장가 슬리피지 측정
  - 레짐별 성과 일관성
  - 시그널 빈도가 Paper와 유사한지

  통과 조건:
  - Fee-adjusted PF >= 1.2
  - 평균 슬리피지 < 0.3%
  - 레짐별 WR 편차 < 20%p

--------------------------------------------------
4.3 Stage 3: Execution Validated → LIVE 승인
--------------------------------------------------

  Shadow Live 통과 + 22B 최종 승인

  승인 전 22B 체크리스트:
  □ Stage 1 + 2 기준 모두 충족 확인
  □ 현재 레짐과 전략의 regime_filter 일치
  □ 기존 LIVE 전략과의 상관 노출 확인
  □ 초기 포지션 사이즈 결정
  □ 롤백 조건 사전 설정 (예: 1주 PF < 0.8이면 PAUSE)

--------------------------------------------------

PART 6: OPERATIONAL FAILURE SCENARIOS
==================================================

--------------------------------------------------
6.1 API Timeout
--------------------------------------------------
  감지: httpx timeout exception
  즉시 조치: 2회 재시도 (3초 간격)
  재시도 실패: 시스템 모드 → LIMITED
  복구: API 정상 확인 후 자동 복귀
  수동 개입: 불필요 (자동)

--------------------------------------------------
6.2 Partial Fill
--------------------------------------------------
  감지: 주문 응답의 executedQty < requestedQty
  즉시 조치:
    - 체결된 수량에 대해서만 SL/TP 부착
    - 미체결 수량 → 취소 또는 잔여 주문 유지 (config)
  기록: partial_fill 태그로 DB 기록
  수동 개입: 불필요

--------------------------------------------------
6.3 DB Write Failure
--------------------------------------------------
  감지: sqlite3 exception
  즉시 조치:
    - 메모리 큐에 임시 저장 (최대 100건)
    - DB 복구 시 flush
    - 주문 자체는 계속 실행 (주문 > 기록)
  3분 이상 지속: OBSERVE 모드 + 알림
  수동 개입: DB 파일 확인

--------------------------------------------------
6.4 Server Restart
--------------------------------------------------
  감지: 프로세스 시작 시 restart_flag 확인
  즉시 조치: Section 3.4 Restart Recovery 절차
  수동 개입: 불일치 발견 시에만

--------------------------------------------------
6.5 WebSocket Stale Data
--------------------------------------------------
  감지: 마지막 수신 타임스탬프 > 60초 경과
  즉시 조치: WebSocket 재연결 시도
  재연결 실패: REST API fallback
  REST도 실패: BLOCKED + 알림
  수동 개입: 네트워크 확인

--------------------------------------------------
6.6 Duplicate Order Risk
--------------------------------------------------
  감지: signal_id 중복 체크 (Section 3.2)
  즉시 조치: 두 번째 주문 자동 거부 + 경고 로그
  수동 개입: 불필요

--------------------------------------------------
6.7 Cloudflare Tunnel Down
--------------------------------------------------
  감지: Dashboard 접속 불가 (외부에서)
  영향: 매매 실행에 영향 없음 (Bot은 로컬)
  즉시 조치: Telegram으로 상태 알림 계속 수신
  복구: cloudflared 자동 재시작 (서비스 등록 시)
  수동 개입: 장시간 미복구 시 수동 재시작

--------------------------------------------------
6.8 Exchange Maintenance / Outage
--------------------------------------------------
  감지: HTTP 503 또는 거래소 상태 API 확인
  즉시 조치: BLOCKED + 신규 주문 전면 차단
  기존 포지션: SL이 거래소 서버에서 관리되므로 유지
  복구: 거래소 정상 확인 후 자동 복귀
  수동 개입: 장시간 시 포지션 수동 확인

--------------------------------------------------
7.0 운영 헤더 바 (최상단, 항상 표시)
--------------------------------------------------

  시스템 모드:    ACTIVE / LIMITED / OBSERVE / BLOCKED
  거래소 연결:    Binance OK / Hyperliquid OK
  오늘 P&L:       +$22.4 (+1.8%)

  "지금 시스템이 안전한가"를 1초에 판단.

--------------------------------------------------
7.1 Panel 1: Indicators (지표)
--------------------------------------------------
  내용: 종목별 EMA, RSI, ATR, VWAP, 펀딩, OI, 볼륨
  레짐 판정 근거 지표값도 함께 표시
  액션: 종목 클릭 → 상세 차트 페이지

--------------------------------------------------

PART 8: STRATEGY LIFECYCLE (v1.2)
==================================================

  DRAFT → PAPER → STAT_VALIDATED → SHADOW_LIVE
    → EXECUTION_VALIDATED → LIVE → PAUSED → RETIRED

  단계              누가 전환     판단 기준
  ────              ──────────    ──────────────────────
  DRAFT             22B          코드 작성 완료
  → PAPER           22B          코드 리뷰 완료
  → STAT_VALIDATED  Bot 자동     Section 4.1 기준 충족
  → SHADOW_LIVE     22B          Stat 통과 확인 후 진행
  → EXEC_VALIDATED  Bot 자동     Section 4.2 기준 충족
  → LIVE            22B 최종승인  Section 4.3 체크리스트
  → PAUSED          Bot/AI 추천  연속손실3, PF<0.8 등
                    + 22B 확인
  → RETIRED         22B          4주 PAUSED, 개선 없음

  강등 후 재승격:
    PAUSED → 수정 → PAPER (처음부터 재검증)
